## 0. Why this catalog exists — two proven failures, not hypotheses

Both were found by inspecting the live host on 2026-07-17. Neither is a guess.

### Failure 1 — G4+ is false today. The flagship claim broke on a system we built ourselves.

The **Know** project (`/opt/know`, live at know.techinsiderbytes.com:3400) serves **78 images. All 78 were generated by Google Generative AI.** Every one carries a cryptographically signed C2PA manifest, issued by *Google C2PA Media Services 1P ICA G3*, containing in plain machine-readable text:

```
description:       "Created by Google Generative AI."
digitalSourceType: http://cv.iptc.org/newscodes/digitalsourcetype/trainedAlgorithmicMedia
action:            c2pa.edited — "Applied imperceptible SynthID watermark."
```

**The Control Surface has zero awareness of any of it.** `discovered_assets` contains nothing for Know, Gemini, or Google. The shadow AI left a signed confession inside the artifact and our discovery never opened the file.

**Root cause — presence-based discovery is structurally blind.** All 7 probes (`proc-cmdline`, `ss-listen`, `systemctl-list-units`, `docker-ps`, `env-backend-url`, `path-scan`, `env-key-presence`) answer one question: *what AI is running right now?* The Gemini usage ran once — `~/.gemini/history` holds **140 session directories** timestamped 2026-07-14 06:13–06:14, and the images landed 06:20–06:22. By the next scan there was no process, no port, no container, no dependency, and no key in the project. **Nothing to find — except the output, which nobody read.**

Corollary: the images are JPEGs named `.png` (78/78). Extension-based scanning misses them too. Sniff bytes, never trust names.

## 1. SLICE S1–S4 — Shadow AI detection (the operator's ask: detect → analyze → insight → recommend → control)

### 1.1 The taxonomy (design core — build to this)

| Class | What it is | Where the trace lives | Today |
|---|---|---|---|
| **S-R Resident** | running process / port / container / unit | the 7 existing probes | ✅ covered |
| **S-E Ephemeral** | CLI session, one-shot script, build-time generation | CLI history dirs, shell history, credential files, cron/timer defs | ❌ **blind** |
| **S-A Artifact** | the *output* of AI; outlives the system that made it | C2PA / SynthID / EXIF / XMP inside the file | ❌ **blind — proven by Know** |
| **S-D Dependency** | AI SDK in a manifest, provider key in `.env` | package.json / requirements.txt / .env | 🟡 keys only (`env-key-presence`) |
| **S-X External** | SaaS used from a browser; no host trace whatsoever | egress/DNS only | ❌ **out of scope — say so honestly** |

S-X is declared out of scope *in the UI*, not silently omitted. Host-level discovery cannot see a browser tab. Claiming otherwise breaks the trust moat (rail §3).

### 1.2 New probes (all offline, zero network, added to `DISCOVERY_SOURCES`)

Each slots into the existing `DISCOVERY_SOURCES` const map (SPEC 24) so `runDiscoveryScan(source)`, the `scan:discovery:<source>` action, bulk register/ignore, and the 30-day retention sweep all work unchanged.

- **S1 `artifact-provenance`** — the flagship. Walk governed project roots (`DASHBOARD_DISCOVERY_ARTIFACT_ROOTS`, default = registered project paths). For each media file **sniffed by magic bytes, not extension**: read C2PA JUMBF (`jumd`/`c2pa` boxes), XMP, and EXIF. Extract `claim_generator_info`, `digitalSourceType`, actions, `instanceID`, issuer CN. `digitalSourceType == trainedAlgorithmicMedia` ⇒ **AI-generated artifact, high confidence, self-declared**. Emit one asset per *generator identity × project*, not per file (78 files = 1 asset with `fileCount: 78`), or the inbox drowns.
  - Bounded: `maxFiles` (default 5000), `maxBytesRead` per file (read headers/first 256KB — C2PA rides in the header), skip `node_modules`/`.git`/`dist`, wall-clock cap.
  - **Do not verify the C2PA signature chain in v1.** Report `signedBy` as *asserted*, not *verified*. Cert-chain validation needs a trust store and network — that's S1b, and honesty demands we not imply verification we didn't do.
- **S2 `cli-history`** — enumerate `~/.gemini/history`, `~/.claude`, `~/.codex`, `~/.config/opencode`, `~/.aider*`. Report tool, session count, first/last activity. **Metadata only — never read session contents.** (Know's 140 Gemini sessions would have been a standing flag since May.)
- **S3 `credential-file`** — locate provider credential files (`gemini-credentials.json`, `google_accounts.json`, `.aws/credentials`, provider `.env` keys). **Report path, key NAME, mtime, and a 4-char fingerprint of a hash — never the value, never a prefix of the value.** A key on disk that the gateway doesn't know about = an ungoverned path (§1.4).
- **S4 `sdk-dependency`** (this is ULTRAPLAN **M9**, now evidence-backed) — parse `package.json` / `requirements.txt` / `pyproject.toml` in governed roots for AI SDK deps (`@google/generative-ai`, `openai`, `@anthropic-ai/*`, `replicate`, `@ai-sdk/*`, …). Declared-but-unused is still a finding: it means someone intended a path.

### 1.3 Analyze + insight (G7 — AI reasoning before raw data)

New scanner `scanners/shadowAi.ts` correlates probe output into one narrative per finding. The Know case must produce **exactly** this, and it is the acceptance test:

> **78 AI-generated images are serving know.techinsiderbytes.com.** Created by Google Generative AI (self-declared, SynthID-watermarked), generated 2026-07-14 06:20–06:22 from an ungoverned `gemini` CLI session on this host (140 sessions, `~/.gemini/history`). Not in inventory · no owner · no gateway ledger entry · cost unknown and unattributable. From **2026-12-02** these artifacts fall under the EU AI Act content-marking obligation. *Recommend: register as a governed asset, assign an owner, route future generation through the gateway (§1.4), and confirm marking is preserved on publish.*

Correlation rules: artifact mtime ↔ CLI session window ↔ credential file ↔ serving surface (which registered service's document root contains the path). Attribution is stated with a confidence and **`temporal-correlation` is labeled as correlation, never as proof**. We infer that the CLI made the images because the timestamps line up (06:13–06:14 sessions → 06:20–06:22 files); we do not *know* it. Say which.

New detectors → insights:
- `shadow-ai:ungoverned-generator` (high) — AI artifacts whose generator has no gateway route.
- `shadow-ai:ungoverned-credential` (high) — provider key on disk not attached to a gateway key.
- `shadow-ai:ephemeral-tool` (medium) — CLI tool with sessions but no governed identity.
- `shadow-ai:unmarked-ai-content` (medium→high after 2026-12-02) — see §3.
- `shadow-ai:unowned-ai-asset` (medium) — ULTRAPLAN **M10**, now with real targets.

### 1.4 Control (the part that actually closes it)

**Architectural statement: every shadow-AI finding is a gateway-bypass finding.** Shadow AI is precisely AI that reached a model without passing our chokepoint. Detection is not the fix — detection finds the bypasses; **capture** is the fix.

Actions (all through the existing risk-tier executor + `action_audit`):
- `register:shadow-asset:<id>` (low) — reuses `registerAssetCore` (SPEC 24). Owner **required** (M10).
- `govern:generator:<tool>` (medium, confirm+reason) — mint a scoped gateway key for the tool (M19 per-agent identity), write its config to point at the gateway, so its calls land in `gateway_calls` from then on. **The bypass becomes governed and the ledger sees it forever.** This is the only action here that permanently fixes anything.
- `quarantine:credential:<path>` (high, approval-gated) — flag an ungoverned key for rotation. **v1 reports and links to the rotate flow; it does not delete or move key material.** Auto-deleting credentials on a live box is how you take the stack down.
- Ignore-with-reason (low) — exists; deliberate non-governance is a valid, audited answer.

**Never auto-appliable.** No shadow-AI action enters `SAFE_AUTO_ACTIONS`; none gets an `AUTO_ROLLBACK_AFFORDANCES` entry in v1. Touching credentials and generator configs automatically is exactly the class of action the promotion review (2.4) exists to refuse.

### 1.5 Schema

```sql
-- reuse discovered_assets (kind: 'ai-artifact' | 'ai-cli' | 'ai-credential' | 'ai-dependency');
-- fingerprint_json carries the class-specific payload. New table only for artifact detail:
CREATE TABLE IF NOT EXISTS ai_artifact_provenance (
  id            TEXT PRIMARY KEY,
  tenant_id     TEXT NOT NULL,
  asset_id      TEXT,              -- FK-ish → discovered_assets.id
  path          TEXT NOT NULL,
  sha256        TEXT NOT NULL,     -- content identity; survives rename/extension lies
  detected_type TEXT NOT NULL,     -- sniffed, not extension
  claimed_type  TEXT,              -- what the extension claimed (Know: png vs JPEG)
  standard      TEXT NOT NULL,     -- 'c2pa' | 'synthid-declared' | 'exif' | 'xmp'
  generator     TEXT,              -- "Google Generative AI"
  source_type   TEXT,              -- IPTC digitalSourceType URI
  signed_by     TEXT,              -- issuer CN — ASSERTED, not verified (v1)
  signature_state TEXT NOT NULL DEFAULT 'unverified'
                CHECK (signature_state IN ('unverified','valid','invalid')),
  instance_id   TEXT,
  created_ts    INTEGER,
  first_seen    INTEGER NOT NULL,
  last_seen     INTEGER NOT NULL,
  raw_json      TEXT NOT NULL
);
CREATE UNIQUE INDEX IF NOT EXISTS idx_ai_artifact_tenant_sha ON ai_artifact_provenance (tenant_id, sha256);
```

## 2. SLICE S5–S8 — Model truth (`/ratings` unparked, R5 revived)

### 2.1 The correction

R5 and P5.2 were parked because `model_eval_runs` is empty. **Wrong data source.** Kilo's entire thesis — *rank on real usage, not synthetic benchmarks* — is the argument against how we parked it. We hold **9,471 `gateway_calls`** (`logical_model`, `resolved_model`, `backend`, `latency_ms`, `cost_estimate_usd`, `success`, `error_class`, `caller`) and **426 `builder_passes`** (`model`, `provider`, `status`, `failure_class`, `phase`, timings). That is an outcome corpus. **It is better than an Elo, because it is fit-for-*our*-workload, which is the only question that matters.**

A synthetic eval tells you a model is smart. `builder_passes` tells you it finished the job.

### 2.2 S5 — first-party outcome leaderboard (Kilo-shaped)

`server/ratings/outcomes.ts` — no new tables in v1; a view + aggregate over existing rows.
- **Task axis** (`builder_passes`): success rate, avg duration, failure-class mix per model. Denominator excludes `canceled` (17 rows) — a cancel is not a model failure.
- **Traffic axis** (`gateway_calls`): success %, p50/p95 latency, error-class mix, cost per model.
- Minimum-sample rail: **`n < 5` renders "insufficient data", never a percentage.** A 100% from 3 passes (`alibaba/qwen3-coder-plus`) is noise wearing a crown.
- Recency-bounded windows (7d / 30d / all) — the 80.8%-all-time vs 0%-last-7d split on `coding-go-minimax-m3` is the whole point: **an all-time average hides a dead model.** Default view = 7d.

### 2.3 S6 — the waste metric (the differentiator; do not skip)

Total spend across those 9,471 calls over two months: **$0.4551.** Denominated in dollars, `/cost` reads forty-five cents forever and looks like nothing. **For a free-first stack the currency is not dollars — it is wasted attempts and wasted wall-clock.** `zen-deepseek-v4-flash-free` costs $0.00 and burns chain latency on 191 calls, 66 of them this week.

> ⛔ **BLOCKED on `REPAIR_PLAN_MODEL_ROUTING.md` R0.** All three metrics below require grouping calls into requests, and **9,466 of 9,471 calls carry no `trace_id`**. They are uncomputable today. R0 stamps the id; give it ~48h of accumulation before building this slice. (Same blocker kills ULTRAPLAN **M21 lineage**, which is scoped as "a JOIN + UX slice — all the data exists." It does not.)

Define and surface (post-R0):
- `wasted_attempts` = calls to routes with `success=0`, by model, by chain position.
- `wasted_seconds` = Σ `latency_ms` on failed calls before the chain reached a success.
- `time_to_first_success` = wall-clock from chain entry to first successful response — **the number the operator actually feels.**

Nobody in the §11.4 competitive set (CloudZero/Amnic/Vantage) measures this, because their buyers pay per token, so dollars *are* their waste. Free-first stacks have no such metric and we are the only ones positioned to define it. This is R2's affordability story finally expressed as a number that moves.

### 2.4 S7 — dead-route detector + chain pruning (fixes Failure 2, and is the self-tuning loop)

New scanner `scanners/modelRoutes.ts`:
- `models:dead-route` (high) — route with `n ≥ 5` and **0% success** over the window. Live targets today: `zen-minimax*` (×4), `zen-kimi*` (×2), `openrouter-minimax-free`, `zen-deepseek-v4-flash-free`, `zen-hy3-free`, `groq-groq-compound`, `groq-groq-compound-mini`.
- `models:probe-only-route` (medium) — configured, probed once, never carried real traffic, still sitting in fallback chains. **Signature: exactly N calls, all on one day, all failed.** This is the class that rots a config silently.
- `models:degraded-route` (high) — was healthy (`coding-go-minimax-m3` 80.8% all-time), now 0% over the recent window. Distinct from dead: **this one is worth fixing, not pruning** (11 `auth` failures ⇒ likely an expired opencode-go credential — investigate before deleting a route that earned 526 calls).
- `models:orphaned-capability` (medium) — **a model that works on one provider with no working route on another.** Today: `opencode-go/kimi-k2.6` scores 82.1% on 28 builder passes, yet the config's only kimi routes are the two dead `zen-kimi-*` entries. Recommend adding `coding-go-kimi-k2-6 → hosted_vllm/kimi-k2.6`. **This detector is the operator's exact question #2, automated.**
- `models:broken-paid-fallback` (high) — `github-gpt41` at 3% with 159 `auth` failures, fanning out to a ~40-model chain, while being the only route that costs money. Per `feedback_free_models_first`, paid is last resort — but a *broken* last resort is a chain that ends in nothing.

**Wire the recommendation into the existing A4c chain-preview** (`GET /api/models/chain-sync`, commit `ddf76cb`) — which is already read-only and already emits a corrected `router_settings.fallbacks` block plus the operator-run apply command. Dead routes drop out of the emitted chain; orphaned capabilities get proposed in. **Per the standing operator decision, the Control Surface still never writes `/etc/litellm/config.yaml`.** Copy-button, not execute-button. This is the parked "self-learning remediation loop" made real *without* handing the config a pen.

### 2.5 S8 — external axis (cold-start only) — **use LiveBench, not Arena**

A purely first-party leaderboard is **empty on every fresh host** — zero calls, zero passes, zero day-one value, and our own fresh-host gate would rightly render honest-empty on `/ratings`. That is the G9 problem the external axis exists to solve, and **that is its only job.**

**Source decision (settled 2026-07-17): [LiveBench](https://livebench.ai/#/).** It beats LMArena on every axis that matters to us:

| | LiveBench | LMArena / Chatbot Arena |
|---|---|---|
| Scoring | **objective ground truth, no LLM judge** | crowd preference (Elo) |
| Contamination | **new questions monthly**, ~1/6 refreshed, held private 1 month | n/a |
| Categories | **Coding, Instruction Following**, Reasoning, Math, Data Analysis, Language | one general chat score |
| Machine-readable | **`download_leaderboard.py` + HuggingFace (`livebench` org)** | unclear |
| License | **Apache-2.0 + MIT** — redistribution permitted with attribution | unconfirmed |

**The licensing blocker is resolved.** LiveBench is Apache-2.0 (inherited from **FastChat** — itself an LMSYS project, so the operator's first and last link close the same loop) plus MIT (LiveCodeBench). Both permit redistribution with attribution ⇒ **we may legally bundle a dated snapshot in the repo.** Preserve the license notices and attribute in `/about` + `/ratings`.

Why it's the right prior specifically for us: **`Coding` and `Instruction Following` are the two categories that plausibly predict builder-pass success** — and instruction-following is the known failure mode of our own stack (`project_builder_backend_resilience`: agentic passes need tool-calling models; nemotron-3-ultra-free at 39.3% with 1909s averages is an instruction-following/idle-timeout failure, not an intelligence failure). Arena Elo measures whether strangers liked the chat reply. That has never been our question.

- Bundle a dated snapshot; `GET /api/ratings/external` serves it **offline** (CSP-clean, nothing leaves the box); optional refresh action re-runs `download_leaderboard.py`; staleness surfaced like `model-health.json`.
- **Name resolution is the real engineering problem, not licensing.** Our routes are provider aliases (`coding-go-minimax-m3`, `zen-big-pickle`, `zen-north-mini-code-free`) and LiveBench lists upstream model names. Build an explicit alias map; **unmatched routes render "no external rating" — never guess, never fuzzy-match.** Expect partial coverage and say so per row. Several of our routes will have no LiveBench entry at all; that is fine and must look fine.
- **Label permanently:** LiveBench measures the model. We measure the model *on our repo, through our provider, under our timeouts*. Present external as the cold-start prior that first-party data **replaces** as it accrues, with a visible "now using your data (n=…)" switchover.

**S8b — the experiment worth running (cheap, and interesting either way).** Once both axes exist, correlate LiveBench `Coding`/`Instruction Following` against our own 426 builder passes. If it predicts, the prior is validated and fresh hosts get a genuinely useful day-one ranking. **If it doesn't predict, that is a real finding and a publishable one** — "public benchmarks don't predict agentic outcomes on a real repo, here's 426 passes of evidence" is exactly the kind of claim this product is built to make, and it would be a marketing asset in the class of `CASE_STUDY_CLOSED_LOOP.md`. Do not prejudge the direction; report what the data says.

### 2.6 S10 — provider roster reconciliation (declared vs observed)

Source: **[awesome-free-llm-apis](https://github.com/mnfst/awesome-free-llm-apis)** — **CC0-1.0** (public domain, zero restrictions ⇒ bundle freely) with a machine-readable **`data.json`** carrying base URLs, model names, context windows, modalities, and **declared rate limits (RPM/TPD)** across 12 provider APIs + 15 inference platforms.

> **Note the reversal, deliberately:** §2.6's table refuses `free-ai-tools` as an importer *because it is README-only*. That reasoning inverts here — structured `data.json` + CC0 is exactly the machine-readable, unencumbered source the earlier refusal was asking for. Same test, opposite answer. Don't let the earlier line be quoted against this one.

**The gap this fills — we learn every limit by hitting it.** `829 of 9,471 calls (8.8%) were rate-limited`, and we hold *observed* 429s (`zen-deepseek-v4-flash-free` 135, `zen-big-pickle` 135, `github-gpt41` 86, `coding-go-minimax-m3` 26 …) but not a single *declared* limit. Declared limits turn an empirical crash-into-the-wall into scheduling.

- **S10a — breadth gap detector.** Configured provider prefixes today: `zen` 44, `openrouter` 36, `cf` 27, `groq` 13, `nvidia` 9, `github` 5, `coding-go` 5, `cerebras` 3, `gemini` 3. The roster lists free-tier providers we carry **no route to at all** — Cohere, Mistral, Zhipu, SambaNova, HuggingFace, Aion Labs. Per `feedback_fallback_breadth` (breadth = resilience, interleave providers), each is a candidate chain member. Detector: `models:missing-provider` (low) — proposes, never auto-adds.
- **S10b — declared-vs-observed reconciliation** (the novel bit). Join declared RPM/TPD against observed `error_class='rate_limit'` per route:
  - 429s **above** the declared limit ⇒ *we are over quota* — throttle or space the chain. Fixable by us.
  - 429s **well under** the declared limit ⇒ *the provider silently cut us* — the roster's own footnote records exactly this ("Groq rate limits were reduced in 2026"). Not our bug; re-plan capacity.
  - Declared limit with **zero** observed traffic ⇒ unused free capacity sitting idle while other chains starve.
  - Detector `models:quota-mismatch` (medium) with the distinction stated in the finding — "you're over" and "they cut you" demand opposite responses, and conflating them is how a routing config rots.
- **S10c — config rot at scale.** `zen` holds **44 entries**, the largest share of the config, while Zen now serves ~2 free models (`project_gpu_off_free_cloud`) and its minimax/kimi routes are provably dead (§0 Failure 2). S7's `probe-only-route` detector run across the whole config will quantify the rot; S10 gives the replacement candidates so pruning doesn't shrink breadth.

**Honesty rail — the roster is a hint, never truth.** It is *manually maintained* via community PRs (49 commits). Declared limits are a **claim**, possibly stale the day we read it. **Observed data always wins**; the roster only proposes candidates and explains 429s we already measured. Same doctrine as LiveBench (§2.5): external sources prime, first-party data decides. Bundle a dated snapshot, surface staleness, never let it silently rewrite a chain.

## 3. SLICE S9 — M12 content marking (falls out of S1 for free; hard date)

ULTRAPLAN **M12** has an external deadline: **2026-12-02**. The S1 C2PA reader *is* the mechanism — same code, different corpus.
