# /autopipeline — Product Plan
> One-line: what this page is and who it's for.

A control panel for the editorial autopipeline, allowing operators to monitor the story queue, manage pipeline state, and manually inject content.

## 1. Today (verified, with file:line)
- **Frontend**: `app/routes/AutopipelinePage.tsx` (✅ solid)
  - Fetches and displays pipeline state from `/api/autopipeline` every 2 seconds (`AutopipelinePage.tsx:18`).
  - Shows whether the pipeline is running or paused, including the reason for the pause (`AutopipelinePage.tsx:112`).
  - Provides buttons to "Pause" and "Resume" the pipeline, which call `POST /api/autopipeline/command` with `cmd: "pause"` or `cmd: "resume"` (`AutopipelinePage.tsx:120, 128`).
  - Displays the current story being processed, with details on its stage, timing, and log file (`AutopipelinePage.tsx:143`).
  - Renders a queue of upcoming stories, showing their topic and vertical (`AutopipelinePage.tsx:179`).
  - Includes a form to manually add a new topic to the queue (`AutopipelinePage.tsx:64`).
  - Includes a form to inject a dossier at a specific stage (`AutopipelinePage.tsx:88`).
- **Backend**:
  - `server/api/autopipeline.ts`: Handles `GET /api/autopipeline` (`autopipeline.ts:11`). It calls an adapter to get the pipeline state (`autopipeline.ts:12`).
  - `server/api/actions.ts`: Handles the `POST /api/autopipeline/command` endpoint via `autopipelineCommandHandler` (`actions.ts:63`). It interacts directly with the autopipeline service running on port 3200.
- **Data Sources**:
  - The autopipeline service at `http://127.0.0.1:3200` is the ultimate source of truth. The Control Surface backend acts as a proxy.
- **Readiness**: ✅ solid. The page accurately reflects the state of the autopipeline and provides the core controls needed for manual operation.

## 2. Gaps, mock & broken parts
- **No Insight Integration for Paused Pipeline**: When the pipeline is paused, a banner is shown on the page. However, no `insight` is created. An operator would have to navigate to this specific page to know about the pause. This was noted as a finding to implement in `server/insights/scanners/ops.ts:147` (`mapPipelineFindings`). This is a gap in discoverability.
- **No Insight for Stuck Stories**: The `ops.ts` scanner has logic to detect stories that are stuck in a stage for over 2 hours (`ops.ts:169`). The frontend displays running stories, but there is no visual indicator if a story is "stuck" or a direct link to the corresponding insight.
- **"View Log" is Inactive**: The UI for the current story shows a "View Log" button (`AutopipelinePage.tsx:168`), but it is commented out and has a `// TODO` note, indicating it's not implemented.

## 3. Goal alignment (G1–G8)
- **G1 Usable & stable**: ✅ The page is stable and usable for its intended purpose.
- **G2 Controllable via GUI**: ✅ Provides essential GUI controls for pausing, resuming, and adding to the pipeline.
- **G3 Complete**: 🟡 Mostly complete, but missing log viewing and deep integration with the insights system.
- **G4 Detects everything**: 🟡 The backend *detects* stuck stories and paused pipelines (via `ops.ts`), but the frontend doesn't surface these detections, and discoverability is low.
- **G5 Findable, readable, actionable**: 🟡 Key pipeline states (like "paused") are not findable in the central `/insights` inbox.
- **G6 Prefer automatic; fall back to a single Apply button**: ✅ The pause/resume actions are clear, manual "Apply" style buttons.
- **G7 AI reasoning BEFORE insights**: 🟡 N/A on this page, but the related insights for stuck stories *should* have AI reasoning.
- **G8 An actual admin center**: 🟡 Like `/newsbites`, it's a good standalone tool but needs to be woven into the fabric of the admin center via the insights system.

## 4. Best-practice research
- **Integrated CI/CD Views**: Tools like Jenkins, GitLab CI, or GitHub Actions provide a unified view of a pipeline. They don't just show the current queue; they provide a visual history of runs, success/failure rates, and average stage times. This helps identify bottlenecks.
- **Direct Log Access**: In any pipeline tool, being able to view the raw logs for a running or completed job is fundamental for debugging. The "View Log" button should open a modal or a new page showing a stream of the log file mentioned in the UI.
- **Alert-driven Triage**: When a pipeline pauses or a job gets stuck, the primary notification shouldn't be a banner on a page you have to visit. It should be a high-priority alert (email, Slack, etc.) generated by the monitoring system, with a deep link that takes you straight to the problem.

## 5. Target design
- **Insight-driven Status**: The "Pipeline is paused" banner (`AutopipelinePage.tsx:114`) should be enhanced.
  - It should include a link to the corresponding insight (e.g., `/insights?focus=ops:pipeline-paused`).
  - The presence of the banner could even be driven by a `useApi` call to check for that specific insight.
- **"Stuck" Indicator**: For the "Currently processing" card and items in the queue, add a visual indicator (e.g., a red warning icon) if an insight exists for a stuck story (e.g., `ops:stuck-story:<id>`). Clicking this indicator should navigate to the insight.
- **Implement Log Viewer**: The "View Log" button should be enabled. Clicking it should open a modal that displays the contents of the log file (e.g., `tail -n 200 <log_file>`). This can be fetched via a new backend endpoint.
- **Visual History**: Below the queue, add a small section showing the last 5 completed or failed stories, with their final status and duration. This provides immediate historical context.

## 6. Features to add (prioritized)
- **MUST**:
  - Implement the "View Log" feature. This requires a new backend API endpoint that can read a specified log file. `GET /api/fs/browse?path=<log_file_path>`.
  - On the frontend, if the pipeline is paused, add a link within the banner to the `/insights?focus=ops:pipeline-paused` insight.
  - Add a visual indicator next to a story if an insight for it being "stuck" exists.
- **SHOULD**:
  - Add a "History" tab or section that shows the last ~20 pipeline runs (both success and failure) with their duration.
  - Add a "Clear Queue" button (with confirmation) for emergency situations where the queue is backed up with bad jobs. This would call `POST /api/autopipeline/command` with `cmd: "clear"`.
- **EXTRA**:
  - Add stage-timing analytics. Display the average time stories spend in each stage (`research`, `write`, `verify`, etc.) to help identify bottlenecks in the pipeline itself. This would require the autopipeline to emit more structured event data.

## 7. Sellable-in-parts
This page could be marketed as a "Job Queue Monitor" or "Pipeline Control" module. For any business running a background job processing system (e.g., for data processing, report generation, media transcoding), this UI provides a simple, clean interface to monitor and manage the queue without needing to SSH into a server or use complex command-line tools. The pitch is "A simple, powerful UI for managing your background job queues."

## 8. Backend work
- **New Endpoint for Logs**: Create a new endpoint, e.g., `GET /api/fs/read-log`, that takes a file path as a query parameter, performs security checks to ensure it's a valid log file path, and returns the file's content. The existing `GET /api/fs/browse` might be adaptable.
- **New Command for Clear Queue**: In `server/api/actions.ts`, enhance `autopipelineCommandHandler` to handle a `cmd: "clear"` to empty the pipeline queue.
- **No Schema Changes**: No database schema changes are required.

## 9. Build slices
1.  **Log Viewer**: Implement the backend endpoint to read log files and wire up the "View Log" button on the frontend.
2.  **Insight Linking**: Update the "paused" banner to include a link to the relevant insight.
3.  **"Stuck" Indicator**: In the `AutopipelinePage`, fetch insights with `domain: "ops"` and check for `sourceKey`s matching `ops:stuck-story:*`. If a match is found for a story in the queue, display a warning icon with a link.
4.  **Clear Queue Action**: Implement the "clear" command in the backend and add the "Clear Queue" button to the UI.
5.  **History View**: Add the UI and necessary backend logic to display recent pipeline run history.

- Verify the "View Log" button opens a modal and successfully displays log content from the backend.
- Verify the "Clear Queue" button, after confirmation, successfully empties the pipeline queue.
