## 4. Best-practice research
- Adopt an "identity governance lifecycle" pattern: access inventory, privileged role review, exception review, approval workflow, and audit proof. Microsoft Entra access reviews emphasize recurring reviews, reviewers, recommendations, and proof of review: https://learn.microsoft.com/en-us/entra/id-governance/access-reviews-overview.
- Use "least privilege with time-bounded elevation": Entra PIM frames privileged access as manage/control/monitor, not permanent ownership: https://learn.microsoft.com/en-us/entra/id-governance/privileged-identity-management/pim-configure.
- Use NIST CSF 2.0's Govern/Identify/Protect/Detect/Respond/Recover framing for the Access module: governance sets policy, detect finds violations, respond applies audited remediation: https://nvlpubs.nist.gov/nistpubs/CSWP/NIST.CSWP.29.pdf.
- For AI operations, align policy work with NIST AI RMF's Govern/Map/Measure/Manage, but implement using the repo's current YAML/JSON policy evaluator first; do not claim OPA until the codebase really adds it: https://www.nist.gov/itl/ai-risk-management-framework.
- "Great" for this page: a reviewer opens Access & Policy and immediately sees access risk score, expiring approvals, owner sprawl, stale secrets, log-only policy exceptions, and one Apply path with audit evidence.

## 5. Target design
- IA: header with Admin Health contribution, Access Risk Score, pending high-risk approvals, stale policy exceptions; tabs become `Overview`, `Users & Roles`, `Policies`, `Approvals`, `Secrets`, `Budgets`, `Retention`, `Audit`.
- Overview: top cards for "Owners", "Pending approvals", "Secrets needing rotation", "Budget cap", "Log-only controls", then an embedded filtered inbox for `domain=security` findings whose `manualPageHref` is `/governance`.
- Users & Roles: searchable user directory from `users` + `governance_role_bindings`; role editor with least-privilege recommendation, "last action" from `action_audit`, and recurring review workflow.
- Policies: show loaded policy file(s), parsed rules, effect, conditions, decision counts, last decision, dry-run test form, diff preview, and Apply-to-write flow. Keep a clear label: "Local YAML/JSON policy evaluator".
- Approvals: show request, required votes, expiry, evidence, requester, linked insight/action, and AI "why this approval exists" before approve/reject.
- Secrets: show inventory with status only, never plaintext; actions: rotate, test reference, revoke/delete, assign owner, set expiry. Use drawer modals, not inline table text.
- Budgets: global and project scopes, cap, warn threshold, current spend, linked budget/security findings, one Apply button for suggested cap.
- AI Inventory handoff: show an `Unregistered AI assets` panel sourced from discovery for assets needing owner/policy/budget assignment; each row links to its finding and supports audited Register, Ignore-with-reason, and Re-scan actions without editing `system.ts`.
- Empty/loading/error: each tab gets a non-deceptive empty state and Retry; no blank tables.
- Mobile: table views collapse to cards, all buttons ≥44px, destructive actions use full-width confirm sheets.
- G7/G6: every finding panel starts with AI summary/root cause/recommended action; safe resolved states are automatic, high-risk mutations open approval, routine review-tier actions use a single Apply with reason.
